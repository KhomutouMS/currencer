## 1. Введение

Данный документ описывает план тестирования десктопного приложения "Currencer". Документ предназначен для команды тестирования и разработки. Цель тестирования - проверка соответствия реального поведения программы и ее ожидаемого поведения, описанного в [SRS в документе](Documentation/Requirements/SRS.md) .

<a name="2-объект-тестирования"/>

## 2. Объект тестирования

В качестве объектов тестирования можно выделить следующие функциональные требования:

* Система аутентификации и регистрации пользователей
* Управление валютными парами
* Система уведомлений о курсах валют
* Система кошельков и транзакций
* Создание отчетов
* Получение актуальных курсов валют через API

<a name="3-атрибуты-качества"/>

## 3. Атрибуты качества

### 3.1 Функциональность:
- **Функциональная полнота**: приложение должно выполнять все заявленные функции
- **Функциональная корректность**: приложение должно выполнять все заявленные функции корректно
- **Интероперабельность**: приложение должно корректно работать с внешними API

### 3.2 Удобство использования:
- **Интуитивность интерфейса**: все функциональные элементы пользовательского интерфейса должны иметь понятные названия
- **Время обучения**: основные функции должны быть освоены пользователем за 3 минуты
- **Отзывчивость**: время реакции на действия пользователя должно быть минимальным

### 3.3 Надежность:
- **Отказоустойчивость**: приложение должно корректно обрабатывать ошибки соединения с API
- **Стабильность**: приложение не должно завершать работу аварийно

### 3.4 Безопасность:
- **Защита данных**: данные пользователей должны храниться в защищенном виде

<a name="4-риски"/>

## 4. Риски

К основным рискам можно отнести:

* Отсутствие интернет-соединения для получения данных с API
* Недоступность внешнего API Frankfurter
* Ограничения бесплатного API (лимиты запросов)
* Проблемы с кодировкой при сохранении данных пользователей
* Конфликты доступа к файлам данных при одновременной работе

<a name="5-аспекты-тестирования"/>

## 5. Аспекты тестирования

### 5.1 Система аутентификации
- Регистрация нового пользователя
- Вход существующего пользователя
- Вход как гость
- Валидация вводимых данных
- Сохранение пользовательских данных между сеансами

### 5.2 Управление валютными парами
- Добавление новой валютной пары
- Удаление существующей пары
- Проверка поддерживаемых валют
- Отображение актуальных курсов
- Обновление курсов в реальном времени

### 5.3 Система уведомлений
- Создание уведомления с условием
- Удаление уведомления
- Корректная работа фоновой проверки
- Срабатывание уведомлений при выполнении условий
- Сохранение уведомлений между сеансами

### 5.4 Система кошельков
- Создание нового кошелька
- Добавление валют в кошелек
- Удаление кошельков
- Выполнение транзакций (покупка/продажа)
- Расчет курсов при транзакциях
- Сохранение истории транзакций

### 5.5 Система отчетов
- Создание отчета по выбранному кошельку
- Фильтрация транзакций по дате
- Корректный расчет статистики
- Формирование детального отчета

### 5.6 Работа с внешним API
- Получение актуальных курсов валют
- Обработка ошибок соединения
- Обработка таймаутов
- Корректный парсинг JSON ответов

### 5.7 Нефункциональные аспекты
- Производительность при работе с большим количеством данных
- Удобство навигации между окнами
- Читаемость отображаемой информации
- Корректность сообщений об ошибках

<a name="6-подходы-к-тестированию"/>

## 6. Подходы к тестированию

### 6.1 Ручное тестирование
- Тестирование пользовательского интерфейса
- Проверка бизнес-логики
- Тестирование интеграции с API

### 6.2 Тестовые сценарии
Для каждого аспекта тестирования будут разработаны тестовые сценарии, покрывающие:
- Нормальные случаи использования
- Граничные случаи
- Ошибочные ситуации

### 6.3 Тестовые данные
Будут подготовлены тестовые наборы данных, включая:
- Тестовых пользователей
- Наборы валютных пар
- Тестовые кошельки с различными валютами
- История транзакций для тестирования отчетов

<a name="7-представление-результатов"/>

## 7. Представление результатов

Результаты тестирования будут представлены в следующих документах:

1. **Матрица трассировки требований** - соответствие тестов требованиям SRS
2. **Отчет о дефектах** - обнаруженные ошибки с приоритетами
3. **Отчет о тестировании** - общие результаты тестирования
4. **Рекомендации** - предложения по улучшению

Все отчеты будут включать:
- Описание тестируемой функции
- Ожидаемый результат
- Фактический результат
- Статус теста (пройден/не пройден)
- Серьезность обнаруженных дефектов

<a name="8-выводы"/>

## 8. Выводы

Данный тестовый план позволяет протестировать основной функционал приложения "Currencer". Успешное прохождение всех тестов не гарантирует полной работоспособности на всех версиях Windows, однако позволяет полагать, что данное программное обеспечение работает корректно в заявленных условиях.

Особое внимание при тестировании следует уделить:
- Работе с внешним API в условиях нестабильного соединения
- Корректности финансовых расчетов
- Сохранности пользовательских данных
- Удобству интерфейса для различных сценариев использования

Тестирование должно проводиться на различных версиях Windows для обеспечения совместимости.
